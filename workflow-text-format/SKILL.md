# 0. 目的

- **说明**：帮助编写或生成 **结构清晰、易于扫读与跟随** 的工作流类 Markdown；具体可执行规则以本文为准，并随与各锁定脚本的同步机制增量更新。

# 1. 双向锁定（本 Skill ↔ 多个工作流脚本）

- **关系**：本 Skill 与 **任意数量** 的工作流脚本之间可建立 **一对多** 双向锁定。**纳入** 同一套规则的每个脚本，须在自身的 **「生成规范」**（或同等节）中 **明确声明**三件事（可用条目概括，**不必**复述下文条文）：（1）与本 Skill 的 **双向锁定** 及本 **SKILL.md** 路径（文首 YAML 载明路径者，正文可互指）；（2）工作流正文与格式的 **整体逻辑以本 Skill 为准**；（3）**树形次级** 的新增、调整或加深嵌套 **须遵守** **`# 1`** 本条下 **树形次级增改** 的规则与自检。未作上述声明的脚本 **不视为** 与本 Skill 锁定。

- **树形次级增改**：锁定脚本中 **新增、调整或加深嵌套** 树形结构（`├─`、`└─`、`│`）时，须 **先查阅** **`# 2. 层级标识`** 下 **树形次级** 全文与 **树形次级案例**；**定稿前**按下列自检逐项通过。
    ├─ **末支**：同一父节点下最后一子须为 `└─`，不得以 `├─` 作为该层收尾。
    ├─ **竖向延续**：父行为 `├─` 且同层仍有后续兄弟时，其间子树各行左侧须有 `│` 贯通至该子树闭合。
    ├─ **内部空行**：同一二级项目之下的 **树形次级** 子树内无空白行（含代码围栏与示例块前后）；详见 **`# 2` · 树形次级** · **内部空行**。
    └─ **同构**：缩进、符号列与 **树形次级案例** 一致后再保存。

- **操作约定**：
  ├─ **脚本遵循 Skill**：对 **任一** 已锁定脚本的正文增删改，须符合 **本 SKILL.md** 中已存在的全部规则。
  ├─ **新格式回写 Skill**：在 **任一** 锁定脚本中需要引入 **本 Skill 尚未记载** 的文本格式时，应将新规则 **写入本 SKILL.md**；若与存量 **无冲突**，直接增补。
  ├─ **冲突时停手并询问用户**：若提示词或拟议规则与 Skill **存量冲突**，**不得**擅自修改 Skill 或相关脚本中的冲突条款；须向用户 **说明冲突内容**，请用户在可选方案中 **选定一种**，或说明 **如何兼容** 后再改。
  └─ **Skill 自身**：对本 SKILL.md 正文的增删改，同样须遵守下文 **`# 2`**～**`# 4`** 所列规则（与各锁定脚本同一套显示规范）。

- **示例（当前已锁定）**：`misc/RAG/open-source-project-deepdive.md`；`/Users/yangjiachen/xbx/program/bmac_dev/bmac-dev-workflow.md`（`bmac_dev` 项目根目录下的 BMAC 数据中心工作流）；`/Users/jiachenyang/Documents/plst/overview.md`（`plst` 项目根目录下的力量举策略开发 Agent 总体架构总纲）；`/Users/jiachenyang/Documents/plst/datacenter.md`（`plst` 数据中心模块开发文档）；`/Users/jiachenyang/Documents/coin_backtest/cc/claude-code-domestic-llm-workflow.md`（`coin_backtest` 仓库 `cc/` 下，Claude Code 接入国产 LLM 流程）；`/Users/jiachenyang/.claude/skills/claude-settings/SKILL.md`（Claude Code 全局 Skill，权限体系与 settings.json 优先级规则）。

# 2. 层级标识

- **适用范围**：下文条文 **同时适用于**（1）本文件 `workflow-text-format/SKILL.md` 的正文；（2）**所有** 已在 **「生成规范」**（或同等节）中按 **`# 1` 关系** 完成声明、与本 Skill 双向锁定的工作流脚本。**具体符号、层级与对象标识等可执行条文** **只在本节及以下维护**；各脚本 **生成规范** 中 **不展开** 树形细则，**树形次级增改** 的操作入口与自检仍以 **`# 1`** 为准。更新时以本节为准并视需要调整各锁定脚本；若处于上条「冲突待用户裁决」状态则暂缓同步。

- **一级项目**：
  ├─ **语法**：行首单个 `#`；**不得**使用 `##`、`###` 等 Markdown 次级标题语法。
  ├─ **形态与索引**：须为 **`# x. 名称`**，其中 **`x` 须** 为从 **`0` 起** 的非负整数（`0`、`1`、`2`、…），**`.` 后空一格**再接 **名称正文**；依出现顺序自 `0` 起顺号，同一文档内体例**须** 一致。**不得** 使用小写拉丁字母 `a`、`b`、`c`、… 作为 **一级** 项目索引（含历史文档迁移时，应**改为** 数字体例后与本条对齐）。
  ├─ **文件名**：文件名即 **一级项目**显示名，正文不写与文件名重复的 **主项目行**。
  └─ **YAML 后起篇**：YAML `---` 闭拢后，正文以 **第一个** 符合本条的 `# x. …` **一级项目**起篇（`x` 为 **`0` 起** 的整数，见上款）。

- **二级项目**：
  ├─ **形式与禁止**：每个 **一级项目** 之下，正文 **必须** 仅用 **`- **加粗标签**：正文`** 组织（行首 `-` + 加粗标签 + 冒号 + 正文）；**禁止**裸段落或单独一行的 **`**加粗标签**：……`**。
  ├─ **与 Markdown**：**二级项目** 专指上述列表项，与 Markdown 二级标题 `##` 无关（本规范禁用 `##`）。
  └─ **相邻间隔**：同一 **一级项目** 下，**相邻两条**二级项目（均为行首 `- **标签**：` 的独立列表项）之间 **须空一行**；该空行 **仅** 出现在两条二级项目之间，**不得**落在任一二级项目之下的 **树形次级** 内部。树形内部的连续排版规则见 **树形次级** · **内部空行**。

- **树形次级**：
  ├─ **写法**：每条 **二级项目** 之下若仍有细分，**一律**用树形符号与缩进展示层级；**不再**使用嵌套的 `-` 无序列表承载次级结构（二级项目本身仍是唯一的 `-` 前缀层级）。
  ├─ **内部空行**：自某条 **二级项目** 之下第一条 `├─` / `└─` / `│` 起，至该二级项目所承载的树形子树结束，**不得**插入空白行（包括树节点行之间、树行与缩进的 fenced code 围栏行之间）。兄弟 `├─`/`└─`、代码示例块均紧接书写。**例外**唯有上款「相邻两条二级项目之间 **须空一行**」——该空行不得夹在树形行中间。
  ├─ **`├─`**：同一父节点下，**本行之后仍有至少一条同父的兄弟行**时使用（即本行 **不是** 该父节点下 **最后一支**）。**禁止**：同父 **仅 1 个子节点** 仍用 `├─`——此时 **必须** 改用下条 **`└─`**。
  ├─ **`└─`**（**强制对齐**）：同一父节点下，**兄弟中的最后一支**；**等价规则**：同父 **仅有 1 个子节点** 时，**必须** 使用 `└─`（唯一子即末子，不存在「单独的中间支」）。**自检**：自根向叶扫读，对每个父节点 **数清直接子女行数**；若 **n=1** 且该行是 `├─`，即 **格式错误**，须改为 `└─` 并 **按下一款** 校正缩进。
  ├─ **单支下探缩进**：当父行已是 `└─` 而仍需 **继续向下** 画孙辈及以下时：**`│` 只继承、不硬加**——若 **更上级路径** 上已有 **须用 `│` 贯通** 的情形（例如某祖先层存在 **`├─` 与后续兄弟**，下行需 **竖线承上**），则 **维持** 既有的 **` │` 列** 与缩进，更深一行 **加宽空格** 对齐到该单支末端，**禁止** 在 **已无同层兄弟** 处 **再叠一列** 多余 `│` 后才接 `└─`（**误排**：`│   └─` 父 接 `│   │   └─` 子；**宜**：`│   └─` 父 接 `│       └─` 子）。若从 **该 subtree 的根** 起 **直至当前父** **全程单支**（**没有任何一层** 因「尚有兄弟」而需要 `│`），则 **不要** 为对齐 **新画** 本不需要的 **`│`**，改用 **仅空格缩进** 下探即可。
  └─ **`│`**：
      ├─ **竖向延续**：在 **`├─`** 支系下、同一父节点多行子节点时，用 **`│`** 列对齐（多行树时对齐用）。
      └─ **`└─` 下同级（易混点）**：当前行若是 **`└─`**，即 **同父末子**。其 **直接子女** 若 **无**「并排多兄弟」需 **`│` 竖向贯通**，则 **勿** 在 **`└─` 父行下** 再叠 **一整列 **`│`** 后才接 **`├─`/`└─`**（常见误排：`│   ├─`）；**宜** **加深空格** 后直接 **`├─`/`└─`**，**父级左侧不必再竖向画 `│`「跟着下探」**（与上款 **单支下探缩进**「全程单支不新画 `│`」同旨）。**仅**约束 **本条 `└─` 之父** 的 **直接子女**；**他层** **`│`** 仍依 **竖向延续** 等规则另判。

- **树形次级案例**：下列为 **节选重构**（两类树形合示于 **同一棵示意树**；标签、语义可替换）。**加粗辨析**：锁定脚本 `misc/housing_buy/test.md` 中，仅行首 **二级项目**（`- **标签**：`）使用 Markdown 加粗；**树形次级**各条条目标题（紧跟二级之下的 `├─` / `└─` 标签文字）**不使用**加粗，与下文示意树（节选，仍含示范用加粗）不要求逐字一致，**竖向延续、末支、单支下探**等规则相同。**竖向延续**：如 `misc/housing_buy/test.md`「**一、购房与持有成本**」自「买方税费」起——**父行若为 `├─` 且同层仍有后续兄弟**，中间子树须 **`│` 竖向延续**。**末支下再分级**：如「**二、房屋功能**」之 **外界干扰**——当 **二级项目** 在同级已是 **`└─` 末条**，子树 **无须为多兄弟并排** 而 **`│` 贯通**时，**子行仅加深空格 + `├─`/`└─`**，**勿**写成 **`└─` 父** 下接 **`│   ├─`**；**父级左侧不必再竖向画 `│`「跟着下探」**（与 **`└─` 下同级** 一致）。**共通**：**同一父节点下最后一支须为 `└─`**。**之前**仍有「房屋总价」「房屋单价」等未贴出，**之后**亦可接续他节，勿将本树当成全文。
    ├─ **买方税费**：（契税说明与政策依据；语义可替换。）

# 3. 对象标识

- **说明**：工作流正文中出现的各类「对象」——概念、函数、路径、配置等——须用统一的前后缀标识，以便扫读、搜索与格式一致。下列规则与 `# 2. 层级标识` 同时适用。

- **粗体（`**`）**：
  ├─ **用途**：二级项目的加粗标签；正文中需强调的术语、概念或约束。各锁定脚本可对「树形次级」条目标题是否使用 `**` 另作约定；`misc/housing_buy/test.md` 约定树形条目标题不加粗，仅二级项目标签加粗。
  ├─ **写法**：`**文本**`；成对出现。
  └─ **示例**：`- **输入**：`、`**不再**`、`**并列**`、`**阶段1～阶段3**`。

- **斜体（`*`）**：
  ├─ **用途**：函数、方法、可调用对象名称（如入口函数、工具函数、API 调用名）。
  ├─ **写法**：`*名称*`；成对出现。
  └─ **示例**：`*init_history*`、`*update_binance_meta*`、`*create_aiohttp_session*`。

- **反引号（`` ` ``）**：
  ├─ **用途**：文件路径、项目/模块名、目录、类名、配置键、代码字面量、节引用、树形符号等「代码或技术标识符」。
  ├─ **写法**：反引号成对包住文本；多词组合（如路径）整体包住。
  └─ **示例**：`config.py`、`core/utils/file_system.py`、`data/`、`BmacHandle`、`run_time`、`ssl_verify=False`、`# 2. 层级标识`、`├` `└` `│`。

- **中文引号（`「」`）**：
  ├─ **用途**：引用短句、术语或专有表述；与粗体区分——粗体强调，引号引用。
  ├─ **写法**：`「文本」`；成对出现。
  └─ **示例**：`「半成品」`、`「生成规范」`、`「冲突待用户裁决」`。

# 4. 功能模块

- **说明**：功能模块是工作流中可复用的最小业务单元，由 **输入 / 工作流 / 输出** 三部分组成。展示功能模块时须同时遵守 `# 2. 层级标识` 与 `# 3. 对象标识`；以下约定描述**前缀判定**、**结构形态**与**各部分写法**。

- **前缀 `f ·` 与套用**：
  ├─ **判定**：凡某处标题或标签以 **`f ·`** 起头（小写字母 `f`、半角空格、间隔号 **`·`**、再半角空格、后接模块名或阶段名等），该处即 **功能模块入口**；**入口位置不做限定**——可在 **一级项目** 名称行内（如 `# 0. f · 模块名`；`0` 为一级数字索引，见上节 **形态与索引**）、**二级项目** 标签（如 `- **f · 模块名**：`）、或 **树形次级** 任一行（如 `├─ **f · 子模块**：`）。**不得**再根据是否具备输入/工作流/输出等结构做启发式推断；**只认 `f ·` 前缀**。
  ├─ **执行**：自该入口起，其下（或紧随其后的子树/后续规范条）须按本节 **结构**、**工作流 · 函数展示**、**输出 · 目录与文件展示** 排版或补全；无 **`f ·`** 的节点 **不适用** 本节专有条目（仍须遵守 `# 2`、`# 3`）。
  └─ **冲突**：若入口写法或展开方式与 `# 2` 或 `# 3` 冲突，**不得**擅自折中；须说明冲突并请用户裁决后再改（**优先**满足 `# 2`、`# 3`，再保留 **`f ·`** 语义）。

- **结构**：
  ├─ **形态**：**不规定**功能模块入口必须独占一条二级项目。自 **`f ·`** 入口向下展开 **输入**、**工作流**、**输出** 三部分即可；展开时用 **树形次级**（`├` `└` `│`）与/或 **二级项目**，以扫读清晰为准。
  ├─ **输入**：列出模块的前置依赖、配置来源、可复用的已有数据等。
  └─ **工作流**：按调用层级展示函数调用链；**输出**：先给出所有输出数据的总目录，再在总目录下树形拆分子目录和文件。

- **工作流 · 函数展示**：
  ├─ **单条格式**：`*函数名*`（`文件路径`）：简介
  ├─ **层级**：可包含一个或多个层级；子函数作为父函数的树形子节点，体现调用关系。
  └─ **可选项**：若某步为可选，前缀 `可选：` 再接上述格式。

- **输出 · 目录与文件展示**：
  ├─ **总目录**：先以 **输出** 标签给出落盘总路径（如 落盘至 `data/`），作为树的根。
  ├─ **子项格式**（区分「子目录」与「叶子文件」）：
  │   ├─ **子目录**（非最终文件名、仅作路径层级）：`子目录名`：简介（说明该目录下大致放什么、用途即可；不写数据类型与列结构）。
  │   └─ **叶子文件**（最终落盘文件名）：`文件名`：数据类型，数据结构简介；类型与结构简介同一行内用逗号或顿号分隔。**数据结构简介**指能定位读写契约的要点，如 DataFrame 的**主要列名**、dict 的键、或 schema 摘要；不必全文贴样例。
  └─ **层级**：子目录下继续用树形拆分至叶子文件；多个叶子文件可与同目录其它文件**同行**列出（文件名之间用顿号），再**另起树行**分别写各自的类型与结构简介（每条一行，仍遵守「叶子」格式）。
